Message 的家规：日志管理

上节课我们介绍了Log类的代码，Log类的主要功能是管理logSegment以及处理日志消息读写等。

其实，上层还有一个日志管理功能，包括加载Log、创建log、删除log、查询log等，同时还提供了几个周期性的任务，分别是：log-flusher（日志刷盘）、 log-retention（日志保留）任务、recovery-point-checkpoint（检查点刷新）任务以及Cleaner线程（日志清理）。这些功能都在LogManager类里实现。

因此，这节课我们主要来分析LogManager类的源码。

LogManager 类

我们先学习重点字段。

重点字段

  • logDirs：log目录集合，在serverproperties配置文件中通过logdirs项指定的多个目录。每个log目录下可以创建多个分区目录，每个Log都有自己对应的目录。LogManager在创建Log时会选择Log最少的log目录创建Log。

  • scheduler：KafkaScheduler对象，用于执行周期任务的线程池。

  • recoveryPointCheckpoints：类型为Map[File,OffsetCheckpoint]。遍历当前每一个目录，为每个目录并创建一个recovery-point-offset-checkpoint文件，已经对应的OffsetCheckpoint对象。

  • dirLocks：类型为Seq[FileLock]，FileLock集合。用来为每个log目录加文件锁。

  • _liveLogDirs：类型为ConcurrentLinkedQueue[File]，log目录的集合。

重要方法

重要方法不少，我们先从创建和加载Log目录开始分析。

createAndValidateLogDirs()

private def createAndValidateLogDirs(dirs: Seq[File], initialOfflineDirs: Seq[File]): ConcurrentLinkedQueue[File] = {

//1.初始化当前存在的日志目录集合

val liveLogDirs = new ConcurrentLinkedQueue[File]()

val canonicalPaths = mutable.HashSet.empty[String]

//2.遍历目录 log.dirs=/data0/log,/data1/log,/data2/log

for (dir <- dirs) {

try {

//offline日志目录不能有这个目录，如果有了就抛出异常。

if (initialOfflineDirs.contains(dir))

throw new IOException(s"Failed to load ${dir.getAbsolutePath} during broker startup")

//3.如果日志目录不存在，就创建

if (!dir.exists) {

info(s"Log directory ${dir.getAbsolutePath} not found, creating it.")

//创建目录

val created = dir.mkdirs()

if (!created)

throw new IOException(s"Failed to create data directory ${dir.getAbsolutePath}")

}

//4.确保目录有读权限。

if (!dir.isDirectory || !dir.canRead)

throw new IOException(s"${dir.getAbsolutePath} is not a readable log directory.")

if (!canonicalPaths.add(dir.getCanonicalPath))

throw new KafkaException(s"Duplicate log directory found: ${dirs.mkString(", ")}")

//5.把创建好的日志目录加到集合里。

liveLogDirs.add(dir)

} catch {

case e: IOException =>

logDirFailureChannel.maybeAddOfflineLogDir(dir.getAbsolutePath, s"Failed to create or validate data directory ${dir.getAbsolutePath}", e)

}

}

if (liveLogDirs.isEmpty) {

fatal(s"Shutdown broker because none of the specified log dirs from ${dirs.mkString(", ")} can be created or validated")

Exit.halt(1)

}

liveLogDirs

}

第一步：初始化当前存在的日志目录集合。

第二步：遍历设定的Log目录集合。例如，log.dirs=/data0/log,/data1/log,/data2/log。

第三步：如果日志目录不存在，就创建。

第四步：确保目录有读权限，如果没有就抛异常。

第五步：把创建好的日志目录加到集合里。

第六步：返回日志目录集合。

Log目录创建成功后要加载，下面就是加载Log目录的方法。

loadLogs()

这个方法的作用是恢复和加载给定数据目录里的所有日志。


private def loadLogs(): Unit = {

info(s"Loading logs from log dirs $liveLogDirs")

val startMs = time.hiResClockMs()

//1.创建线程池数组。

val threadPools = ArrayBuffer.empty[ExecutorService]

val offlineDirs = mutable.Set.empty[(String, IOException)]

val jobs = mutable.Map.empty[File, Seq[Future[_]]]

var numTotalLogs = 0

//2.遍历每个数据目录

for (dir <- liveLogDirs) {

val logDirAbsolutePath = dir.getAbsolutePath

try {

//3.numRecoveryThreadsPerDataDir默认为1.对每个日志目录创建一个线程组成的线程池，并把线程加入threadPools这个线程池组里。

val pool = Executors.newFixedThreadPool(numRecoveryThreadsPerDataDir)

threadPools.append(pool)

//4.上次的节点关闭是否是正常关闭。

val cleanShutdownFile = new File(dir, Log.CleanShutdownFile)

//5.检查.kafka_cleanshutdown文件是否存在，存在则表示kafka正在做清理性的停机工作，然后跳过本文件恢复日志的工作

if (cleanShutdownFile.exists) {

info(s"Skipping recovery for all logs in $logDirAbsolutePath since clean shutdown file was found")

} else {

info(s"Attempting recovery for all logs in $logDirAbsolutePath since no clean shutdown file was found")

brokerState.newState(RecoveringFromUncleanShutdown)

}

var recoveryPoints = Map[TopicPartition, Long]()

try {

//从检查点文件读取topic对应的恢复点offset信息

//文件为recovery-point-offset-checkpoint

//checkpoint file format:

//line1:version

//line2:expectedSize

//nlines:(tp,offset)

//7.载入每个日志目录的recoveryPoints。

recoveryPoints = this.recoveryPointCheckpoints(dir).read()

} catch {

case e: Exception =>

warn(s"Error occurred while reading recovery-point-offset-checkpoint file of directory " +

s"$logDirAbsolutePath, resetting the recovery checkpoint to 0", e)

}

var logStartOffsets = Map[TopicPartition, Long]()

try {

logStartOffsets = this.logStartOffsetCheckpoints(dir).read()

} catch {

case e: Exception =>

warn(s"Error occurred while reading log-start-offset-checkpoint file of directory " +

s"$logDirAbsolutePath, resetting to the base offset of the first segment", e)

}

//8.要加载的日志文件夹。

val logsToLoad = Option(dir.listFiles).getOrElse(Array.empty).filter(_.isDirectory)

val numLogsLoaded = new AtomicInteger(0)

numTotalLogs += logsToLoad.length

//9.日志目录下的每个分区子目录处理生成一个具体job。

val jobsForDir = logsToLoad.map { logDir =>

val runnable: Runnable = () => {

try {

debug(s"Loading log $logDir")

val logLoadStartMs = time.hiResClockMs()

//10.每个日志子目录生成一个job加载log，创建Log对象。根据读取的recoveryPoints，logStartOffsets进行加载，并把log对象加入到logs集合中。

val log = loadLog(logDir, recoveryPoints, logStartOffsets)

val logLoadDurationMs = time.hiResClockMs() - logLoadStartMs

val currentNumLoaded = numLogsLoaded.incrementAndGet()

info(s"Completed load of $log with ${log.numberOfSegments} segments in ${logLoadDurationMs}ms " +

s"($currentNumLoaded/${logsToLoad.length} loaded in $logDirAbsolutePath)")

} catch {

case e: IOException =>

offlineDirs.add((logDirAbsolutePath, e))

error(s"Error while loading log dir $logDirAbsolutePath", e)

}

}

runnable

}

//11.把job交给线程池处理。

jobs(cleanShutdownFile) = jobsForDir.map(pool.submit)

} catch {

case e: IOException =>

offlineDirs.add((logDirAbsolutePath, e))

error(s"Error while loading log dir $logDirAbsolutePath", e)

}

}

try {

//12.等待jobs中的 Runnable 完成。

for ((cleanShutdownFile, dirJobs) <- jobs) {

dirJobs.foreach(_.get)

try {

cleanShutdownFile.delete()

} catch {

case e: IOException =>

offlineDirs.add((cleanShutdownFile.getParent, e))

error(s"Error while deleting the clean shutdown file $cleanShutdownFile", e)

}

}

offlineDirs.foreach { case (dir, e) =>

logDirFailureChannel.maybeAddOfflineLogDir(dir, s"Error while deleting the clean shutdown file in dir $dir", e)

}

} catch {

case e: ExecutionException =>

error(s"There was an error in one of the threads during logs loading: ${e.getCause}")

throw e.getCause

} finally {

threadPools.foreach(_.shutdown())

}

info(s"Loaded $numTotalLogs logs in ${time.hiResClockMs() - startMs}ms.")

}

步骤较多，请你耐心看完。

第一步：创建线程池数组。这个线程组里的线程都是用来存放为每个Log目录保存的线程。也就是说会为每个Log目录分配一个线程来维持Log目录。

第二步：遍历每个数据目录。

第三步：为每个日志目录创建一个线程组成的线程池，并把线程加入threadPools这个线程池组里。线程数为numRecoveryThreadsPerDataDir，默认为1。

第四步：检测上次的节点关闭是否是正常关闭。

第五步：检查.kafka_cleanshutdown文件是否存在，存在则表示kafka正在做清理性的停机工作。因为在启动一个进程的同时另一个进程可能正在做关闭中的请理性工作，这样就不要再进行恢复工作了。

第六步：载入每个日志目录的recoveryPoints。

第七步：要加载的日志文件夹放入到集合logsToLoad里。

第八步：日志目录下的每个分区子目录处理生成一个具体job，等待后面的步骤去异步调用。

第九步：每个日志子目录生成一个job加载log，创建Log对象。根据读取的recoveryPoints、logStartOffsets进行加载，并把log对象加入到logs集合中。

第十步：把job交给线程池处理。

第十一步：等待jobs中的 Runnable 完成。

总体来说，这个方法其实就是根据已经存在的日志目录来创建Log类对象，而且每个Log类对象都是由一个线程异步去创建的。之所以要用多线程去创建log对象是因为这样做能够提升创建的速度，减少启动的耗时。

好，我们现在开始解析logSegment里的异步任务。

startup()

这个方法是从后台启动线程，主要用于刷新和清理日志的功能。代码如下：


def startup(): Unit = {

/* Schedule the cleanup task to delete old logs */

//五个定时调度的任务。

if (scheduler != null) {

info("Starting log cleanup with a period of %d ms.".format(retentionCheckMs))

//第一个任务：启动kafka-log-retention 周期性任务，遍历所有的log，对过期或过大的日志文件执行清理工作。

//清除条件：1.日志超过保留时间。2.日志大小超过保留大小时，会删除最旧的LogSegment，以控制整个Log的大小。

scheduler.schedule("kafka-log-retention",

cleanupLogs _,//定时执行的方法

delay = InitialTaskDelayMs,//启动之后30s开始定时调度

period = retentionCheckMs,//log.retention.check.interval.ms 默认为5分钟。

TimeUnit.MILLISECONDS)

info("Starting log flusher with a default period of %d ms.".format(flushCheckMs))

//第二个任务：启动kafka-log-flusher周期性任务，对日志文件执行刷盘操作，定时把内存中的数据刷到磁盘中。

scheduler.schedule("kafka-log-flusher",

flushDirtyLogs _,//定时执行的方法

delay = InitialTaskDelayMs,//启动之后30s开始定时调度

period = flushCheckMs,//log.flush.scheduler.interal.ms 默认值为Long.MaxValue。也就是说默认不刷盘，操作系统自己刷盘

TimeUnit.MILLISECONDS)

//第三个任务：启动kafka-recovery-point-checkpoint周期性任务，更新kafka-recovery-point-checkpoint文件。

//向路径中写入当前的恢复点，避免在重启的时候重新恢复全部数据。

scheduler.schedule("kafka-recovery-point-checkpoint",

checkpointLogRecoveryOffsets _,//定时执行的方法

delay = InitialTaskDelayMs,//启动之后30s开始定时调度

period = flushRecoveryOffsetCheckpointMs,//log.flush.offset.checkpoint.interval.ms 默认为1分钟。

TimeUnit.MILLISECONDS)

//第四个任务：启动kafka-log-start-offset-checkpoint周期性任务，更新kafka-log-start-offset-checkpoint文件。

//向日志目录写入当前存储日志中的start offset，避免读到已经被删除的日志。

scheduler.schedule("kafka-log-start-offset-checkpoint",

checkpointLogStartOffsets _,//定时执行的方法

delay = InitialTaskDelayMs,//启动之后30s开始定时调度

period = flushStartOffsetCheckpointMs,//log.flush.start.offset.checkpoint.interval.ms 默认值为1分钟

TimeUnit.MILLISECONDS)

//第五个任务：启动kafka-delete-logs周期性任务，清理已经被标记为删除的日志。

scheduler.schedule("kafka-delete-logs", // will be rescheduled after each delete logs with a dynamic period

deleteLogs _,//定时执行的方法

delay = InitialTaskDelayMs,//启动之后30s开始定时调度

unit = TimeUnit.MILLISECONDS)

}

if (cleanerConfig.enableCleaner)

cleaner.startup()

}

首先这五个从后台启动的周期性任务是通过KafkaScheduler这个Kafka基于java的ScheduledThreadPoolExecutor自定义的调度程序。

  • 第一个任务：启动kafka-log-retention 周期性任务，遍历所有的log，对过期或过大的日志文件执行清理工作。（清除条件：1.日志超过保留时间。2.日志大小超过保留大小时，会删除最旧的LogSegment，以控制整个Log的大小。）

  • 第二个任务：启动kafka-log-flusher周期性任务，对日志文件执行刷盘操作，定时把内存中的数据刷到磁盘中。

  • 第三个任务：启动kafka-recovery-point-checkpoint周期性任务，更新kafka-recovery-point-checkpoint文件。向路径中写入当前的恢复点，避免在重启的时候重新恢复全部数据。

  • 第四个任务：启动kafka-log-start-offset-checkpoint周期性任务，更新kafka-log-start-offset-checkpoint文件。向日志目录写入当前存储日志中的start offset，避免读到已经被删除的日志。

  • 第五个任务：启动kafka-delete-logs周期性任务，清理已经被标记为删除的日志。

总结

这节课我给你介绍了日志目录上层管理类LogSegment类，包括重要字段和核心方法，比如创建日志目录、加载日志目录，以及后台异步执行的管理功能，包括：删除过期日志文件，定期刷盘的任务，定期更新恢复点文件，定期更新log start offset文件，定期清理已经被标记为删除的日志。
